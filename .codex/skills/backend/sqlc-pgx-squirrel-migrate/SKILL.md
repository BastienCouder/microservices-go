# sqlc + pgx/v5 + Squirrel + golang-migrate

## Purpose

Use this skill when implementing or refactoring PostgreSQL persistence in Go microservices.

Goals:
- Typed SQL access with `sqlc`.
- Native PostgreSQL performance with `pgx/v5`.
- Dynamic query composition with `Squirrel` only when filters/conditions are variable.
- Schema versioning with `golang-migrate` (separate from `sqlc`).

## Mandatory Rules

- Always use the latest stable versions of dependencies and tools.
- Never put defaults for required env vars in code or Dockerfiles.
- Keep SQL as source of truth in `.sql` files, not inline strings in handlers/usecases.
- `sqlc` is the default path for CRUD/query logic.
- Use `Squirrel` only for truly dynamic WHERE/ORDER/PAGINATION cases.
- Manage schema only through migrations; never auto-create tables at runtime.

## Standard Layout

```text
services/<service>/
  db/
    query/
      *.sql
  internal/adapter/repository/postgres/
    sqlc/
      (generated by sqlc)
    migrations/
      sql/
        000001_*.up.sql
        000001_*.down.sql
```

## Implementation Workflow

1. Create/modify migration SQL (`up` + `down`).
2. Add/modify sqlc query files under `db/query/`.
3. Configure `sqlc.yaml` with PostgreSQL engine and pgx/v5 sql package.
4. Generate typed code with `sqlc generate`.
5. Implement repository adapter on top of generated `Queries`.
6. Use Squirrel for dynamic SQL paths only.
7. Add integration tests against real PostgreSQL.
8. Wire migration execution in startup or dedicated migrate command.

## Minimal sqlc config

```yaml
version: "2"
sql:
  - engine: postgresql
    schema:
      - internal/adapter/repository/postgres/migrations/sql/*.up.sql
    queries:
      - db/query
    gen:
      go:
        package: sqlc
        out: internal/adapter/repository/postgres/sqlc
        sql_package: pgx/v5
        emit_interface: true
        emit_json_tags: false
```

## Squirrel usage policy

Allowed:
- Optional filters (`team_id`, `user_id`, `status`, date ranges)
- Conditional sorting and pagination

Not allowed:
- Rewriting static CRUD already covered by sqlc

## Migration policy

- Use `golang-migrate` with versioned files.
- Keep forward-only mindset for production; down migrations are still required for dev/test.
- Prefer embedded migrations for containerized deploys.

## Quality gates

- `go test ./...` passes
- DB integration tests (build tag `integration`) pass
- `golangci-lint` passes with strict config
